4. 优化建议与实施计划

高优先级建议

为 update_time 字段添加索引

当前所有使用 ORDER BY update_time DESC 的查询均因无索引而执行全表排序，建议立即创建复合索引：

CREATE INDEX idx_cai_api_use_update_time ON ods_telemetry.cai_api_use(update_time);

预计可将排序查询执行时间从13秒以上降至100毫秒以内。

为 api_key 和 response 添加索引

针对高频过滤条件，建议创建以下索引：

CREATE INDEX idx_cai_api_use_api_key ON ods_telemetry.cai_api_use(api_key);
CREATE INDEX idx_cai_api_use_response ON ods_telemetry.cai_api_use(response(255));

注意：对 response（TEXT类型）创建前缀索引，长度255字符，避免索引过大。

限制排序行数

所有 ORDER BY 查询必须添加 LIMIT，避免全表排序：

SELECT * FROM cai_api_use WHERE response LIKE "%服务异常%" ORDER BY update_time DESC LIMIT 100;

即使需要展示更多结果，也应采用分页机制，而非一次性加载全部数据。

中优先级建议

禁止使用 SELECT *

所有查询必须明确指定所需字段，避免加载无用数据：

-- ❌ 不推荐
SELECT * FROM cai_api_use WHERE api_key="xxx";

-- ✅ 推荐
SELECT update_time, user_name, api_key, response FROM cai_api_use WHERE api_key="xxx";

优化 MIN/MAX 查询

当前8次查询均使用 SUBSTRING(CAST(MIN(`full_question`) AS STRING), 1, 1024)，该操作效率低下：

  • CAST和SUBSTRING在聚合前执行，增加计算负担
  • 建议在应用层处理字符串截断，或在数据库中使用视图预处理
  • 若仅需获取最小/最大值，可考虑使用 ORDER BY full_question LIMIT 1 并配合索引

统一表名引用规范

部分查询使用 cai_api_use，部分使用 ods_telemetry.cai_api_use，建议统一为完整库表名，避免歧义和潜在的权限问题。

长期监控建议

建立慢查询监控告警

建议配置数据库慢查询日志监控，当单次查询超过5秒或扫描行数超过100万时触发告警，通知相关开发人员。

定期执行执行计划分析

对高频查询使用 EXPLAIN 分析执行计划，确保索引被正确使用，避免索引失效（如使用函数导致索引无法命中）。

建立SQL评审机制

在代码合并前强制进行SQL审查，重点关注：SELECT *、无索引过滤、无LIMIT排序、大表JOIN等高风险模式。

本次分析揭示了系统中存在严重的慢查询性能问题，核心原因集中在 ods_telemetry.cai_api_use 表缺乏索引、开发人员普遍使用 SELECT * 以及未限制排序行数。这些问题共同导致了高达199GB的数据扫描量和平均6.72秒的查询延迟。
