4. 优化建议

索引优化建议

  1. 为 cai_api_use 表的 update_time 字段创建复合索引：
    CREATE INDEX idx_update_time ON cai_api_use(update_time);
  2. 为 api_key 和 token 字段创建复合索引，优化高频过滤查询：
    CREATE INDEX idx_api_key_token ON cai_api_use(api_key, token);
  3. 为 response 字段创建全文索引（如支持）以优化 LIKE '%服务异常%' 查询：
    CREATE FULLTEXT INDEX idx_response ON cai_api_use(response);
  4. 为 model_name 字段创建索引，优化与 cai_gpu_monitor 表的JOIN操作：
    CREATE INDEX idx_model_name ON cai_api_use(model_name);

查询改写建议

  1. 避免使用 SELECT *，仅选择必要字段，例如：
    SELECT update_time, user_name, latency, api_key FROM cai_api_use WHERE ...
  2. 将 WHERE response LIKE '%服务异常%' 改为使用全文搜索或建立关键词表，避免前导通配符扫描。
  3. 避免对 TEXT 类型字段进行 MIN()、MAX()、SUBSTRING() 等聚合操作，这些操作会强制全表扫描。
  4. 将重复的复杂查询（如计算ndv和null_count）封装为物化视图或定时任务，避免实时计算。

分区策略建议

鉴于 cai_api_use 表数据量巨大（单表超8亿行），建议实施时间分区策略：

  • 按 update_time 字段进行月度分区，例如：2025年12月、2025年11月等
  • 保留最近3个月数据为热数据，历史数据归档至冷存储
  • 分区后，查询仅扫描相关分区，可将扫描数据量减少90%以上
