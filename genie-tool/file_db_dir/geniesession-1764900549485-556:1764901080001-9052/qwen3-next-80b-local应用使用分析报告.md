#### 4. 响应质量维度：隐性指标缺失

当前数据完全缺失**响应成功率、错误率、用户满意度评分、人工修正率**等关键质量指标。例如：  
- 若“合规问答”调用3次，但每次输出均被人工修改，则其有效使用率为0；  
- 若“巡检机器人”调用16次，但15次返回“无异常”，仅1次触发告警，则其价值极高，但调用频次低是因系统稳定，非功能无效。

因此，**调用次数≠使用价值**。一个应用调用10次但每次精准解决关键问题，其价值远超调用100次但90%为无效请求的应用。

建议立即引入以下质量指标采集：  
- **人工干预率**：模型输出被人工修改的比例；  
- **用户反馈评分**：在IDE插件中嵌入“👍/👎”按钮；  
- **任务完成率**：是否成功生成可运行代码、通过测试用例、修复告警。

### （六）潜在风险与系统性问题识别

基于现有数据，可识别出三项系统性风险，若不及时干预，将导致模型资源浪费、用户信任下降、合规隐患。

#### 风险一：**模型调用入口混乱，存在功能冗余**

“IDE-小金灵码”与“小金通-小金灵码”名称高度相似，调用次数均为3次，极可能为同一功能的两个入口。若二者功能一致，则为**严重的产品设计缺陷**，导致：  
- 用户困惑，不知应使用哪个入口；  
- 开发团队维护两套代码逻辑，增加成本；  
- 监控系统无法准确统计真实使用量。

建议立即进行**代码仓库审计**，确认二者是否为同一服务的不同API端点。若为重复，应**立即下线“小金通-小金灵码”**，并将所有流量重定向至“IDE-小金灵码”，同时更新用户文档。

#### 风险二：**低频应用缺乏监控与告警机制**

当前系统仅记录调用次数，未对“连续7天调用<5次”的应用设置**自动告警**。若“合规问答”连续30天调用为0，系统仍保持运行，造成：  
- 无用服务持续占用内存与网络资源；  
- 模型部署包体积膨胀，影响更新效率；  
- 安全审计时发现“僵尸服务”，引发合规风险。

建议建立**应用生命周期管理机制**：  
- **活跃期**：调用≥5次/日 → 正常运行；  
- **观察期**：调用1–4次/日 → 降低资源配额；  
- **冻结期**：调用=0次/日持续7天 → 自动停用，通知负责人；  
- **下线期**：冻结期持续30天 → 永久删除。

#### 风险三：**命名规范缺失，导致模型归属混淆**

“ClaudeCode+GLM”这一命名方式存在严重误导性。其可能引发以下问题：  
- 运维人员误认为该应用调用GLM45，导致成本核算错误；  
- 安全团队误判其为第三方模型调用，触发合规审查；  
- 财务部门按名称计费，误将Qwen调用计入GLM成本。

建议制定**统一命名规范**：  
- **应用名称** = [功能描述] + [模型后缀（可选）]；  
- 若使用Qwen，**不标注模型名**（默认）；  
- 若使用其他模型，**必须标注**，如“代码审查（GLM45）”；  
- 禁止使用“+”、“&”等模糊连接符。
